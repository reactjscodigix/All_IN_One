7 CRITICAL CORRECTIONS APPLIED
================================================================================

1. LEADS TABLE (New)
   ─────────────────────────────────────────────────────────────────────────
   PROBLEM: No formal lead management system
   SOLUTION: Created 'leads' table with full lifecycle support
   
   FIELDS:
   • first_name, last_name, email, phone, company_name
   • source (Website, Referral, Direct, Email, Social, Other)
   • status (New, Contacted, Qualified, Unqualified, Converted)
   • converted_company_id, converted_contact_id, converted_deal_id
   • created_by, created_at, updated_at
   
   API ENDPOINTS: 5
   • GET /api/leads
   • POST /api/leads
   • GET /api/leads/:id
   • PUT /api/leads/:id
   • POST /api/leads/:id/convert
   
   USE CASE: Manage prospect pipeline from first contact to conversion


2. DEAL-CONTACTS JUNCTION TABLE (New)
   ─────────────────────────────────────────────────────────────────────────
   PROBLEM: Deals could only have one contact; real world has multiple stakeholders
   SOLUTION: Created 'deal_contacts' junction table
   
   FIELDS:
   • deal_id (FK to deals)
   • contact_id (FK to contacts)
   • role (VARCHAR 100) - "Decision Maker", "Financial Contact", etc.
   • is_primary (BOOLEAN) - Mark primary contact per deal
   
   API ENDPOINTS: 3
   • GET /api/deals/:dealId/contacts
   • POST /api/deals/:dealId/contacts
   • DELETE /api/deals/:dealId/contacts/:contactId
   
   USE CASE: Handle multiple decision makers, stakeholders per deal


3. ENTITY-FILES ENHANCEMENT (Updated)
   ─────────────────────────────────────────────────────────────────────────
   PROBLEM: Files could only attach to Company, Contact, Deal, Project
   SOLUTION: Added estimation_id and invoice_id columns
   
   NEW COLUMNS:
   • estimation_id (FK to estimations)
   • invoice_id (FK to invoices)
   
   SUPPORTED ENTITIES NOW: 6
   ✓ Companies
   ✓ Contacts
   ✓ Deals
   ✓ Projects
   ✓ Estimations (NEW)
   ✓ Invoices (NEW)
   
   USE CASE: Attach quotes, specs, receipts to financial documents


4. ACTIVITIES UNIVERSAL ENTITY PATTERN (Refactored)
   ─────────────────────────────────────────────────────────────────────────
   PROBLEM: Separate columns for each entity type = NULL values + hard to extend
   SOLUTION: Added entity_type and entity_id columns
   
   NEW COLUMNS:
   • entity_type ENUM('Company', 'Contact', 'Deal', 'Project', 'Lead')
   • entity_id INT
   
   OLD PATTERN (Problem):
   SELECT * FROM activities 
   WHERE contact_id = 5 OR deal_id = 3 OR project_id = 2;
   → Creates NULL values, many columns
   
   NEW PATTERN (Solution):
   SELECT * FROM activities 
   WHERE entity_type = 'Contact' AND entity_id = 5;
   → Clean, efficient, easily extended
   
   BENEFITS:
   ✓ Single query instead of multiple ORs
   ✓ No NULL columns
   ✓ Easy to add new entity types
   ✓ Better performance with indexes
   
   USE CASE: Track activities for any entity without schema changes


5. ENTITY-NOTES UNIVERSAL PATTERN (Refactored)
   ─────────────────────────────────────────────────────────────────────────
   PROBLEM: Same as activities - separate columns per entity type
   SOLUTION: Added entity_type and entity_id columns
   
   NEW COLUMNS:
   • entity_type ENUM('Company', 'Contact', 'Deal', 'Project', 'Lead')
   • entity_id INT
   
   BENEFITS: Same as activities - consistent pattern across CRM
   
   USE CASE: Attach notes to any entity consistently


6. USERS TABLE MANAGER HIERARCHY (Enhanced)
   ─────────────────────────────────────────────────────────────────────────
   PROBLEM: No organizational hierarchy support
   SOLUTION: Added manager relationship and profile fields
   
   NEW COLUMNS:
   • full_name VARCHAR(200)
   • department VARCHAR(100)
   • manager_id INT (FK to users)
   
   SUPPORTS HIERARCHY:
   CEO (manager_id = NULL)
   ├── Sales Director (manager_id = 1)
   │   ├── Sales Manager (manager_id = 2)
   │   │   └── Sales Rep (manager_id = 3)
   │   └── Account Manager (manager_id = 2)
   └── Engineering Manager (manager_id = 1)
       └── Developer (manager_id = 5)
   
   USE CASE: Define org chart, reporting lines, team structure


7. ROLES & PERMISSIONS SYSTEM (New)
   ─────────────────────────────────────────────────────────────────────────
   PROBLEM: No proper RBAC implementation
   SOLUTION: Created role_permissions and user_roles tables
   
   TABLES CREATED:
   • role_permissions
     - role_id FK to roles
     - permission_name VARCHAR(255)
     - module_name VARCHAR(100)
     - can_create, can_read, can_update, can_delete BOOLEAN
   
   • user_roles
     - user_id FK to users
     - role_id FK to roles
     - assigned_by, assigned_at
   
   RBAC FLOW:
   User → User_Roles → Roles → Role_Permissions → Actions
   
   DEFAULT ROLES:
   1. Admin - Full system access
   2. Company Owner - Manage company + contacts
   3. Deal Owner - Manage deals + projects
   4. Project Manager - Manage projects + teams
   5. Client - View-only access
   6. Lead - Restricted access
   7. Employee - General employee access
   
   API ENDPOINTS: 8
   • GET /api/roles
   • POST /api/roles
   • GET /api/roles/:roleId/permissions
   • POST /api/roles/:roleId/permissions
   • PUT /api/roles/:roleId/permissions/:id
   • GET /api/users/:userId/roles
   • POST /api/users/:userId/roles
   • GET /api/users/:userId/permissions
   
   USE CASE: Implement complete role-based access control
